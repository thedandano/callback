pi-apply — Architecture

Python LangGraph MCP server exposing the same 12-tool surface as go-apply. Each of 5 workflow tools advances exactly one graph node via update_state + invoke(None, config). Sessions persist across server restarts via SqliteSaver at ~/.local/share/pi-apply/sessions.db. Walking skeleton: nodes are deterministic stubs. PDF render and ATS survival via Go subprocess bridge deferred.

Workflow Tool Handler pi_apply/server.py · per-tool guard
Three-step guard before every graph advance: (1) validate input (exclusive args, JSON parse), (2) _session_exists(session_id) via graph.get_state(config), (3) assert snap.next == ("expected_node",) for ordering. On pass: graph.update_state(config, delta) then graph.invoke(None, config). No silent fallbacks — every failure returns a typed error envelope with code and retriable.

Non-workflow tools · bypass graph entirely · no session_id required
onboard_user server.py
Accepts resume_content, resume_label, skills, accomplishments, sections. Stub — not stored. Returns "not yet implemented". No graph, no DB write.
add_resume server.py
Accepts resume_content, resume_label, optional sections. Stub. Logs call. Not persisted to data layer.
get_config / update_config server.py
get_config returns live {"log_level", "go_apply_bin"} from env and bridge._BIN. update_config(key, value) stub. Both bypass graph entirely.
preview_ats_extraction server.py
Read-only view into graph state. Calls graph.get_state(config), returns tailored_t1 or resume_content. Does not advance graph state or write a checkpoint.
compile_profile / create_story server.py
Profile compilation and SBI story stubs. Full argument surface accepted for API compatibility with go-apply's tool contract. Return stub messages — not yet connected to a data layer.
All 7 non-workflow tools return plain JSON directly — no session required, no graph state touched, no checkpoint written. They share the same MCP transport but are completely independent of the LangGraph pipeline.
Go subprocess bridge · deferred — wired but not called in skeleton nodes
bridge.py — subprocess transport pi_apply/bridge.py
Binary resolved at import time (module top level — fail-fast): GO_APPLY_BIN env var → shutil.which("go-apply") → EnvironmentError. _BIN: str = _resolve_binary() — type is str, never None.

Two transport functions:
run_pdfrender(args: list[str]) → bytes — renders tailored resume to PDF
run_survival(args: list[str]) → str — ATS round-trip diff, returns JSON with survival_rate

SubprocessError(cmd, returncode, stderr) raised on non-zero exit.
subprocess · stdin/stdout · args list
go-apply binary ~/.local/bin/go-apply
Go binary. Exposes pdfrender subcommand (reads resume text, writes styled PDF bytes to stdout) and survival-diff subcommand (renders PDF, extracts text, diffs against original JD keywords, returns survival_rate metric surfacing silent ATS data loss).
Gap: finalize node should call run_pdfrender(tailored_t2) to produce a styled PDF, then run_survival(pdf_bytes) to compute ATS keyword survival rate. This is the primary gap between the walking skeleton and a production-ready system. The bridge is fully wired and tested — only the node call sites are missing.

"Isolated" means the component can be replaced by editing one file with no cascading edits. The workflow tool handlers in server.py are the orchestration hub — they intentionally coordinate session validation, state injection, and graph invocation in sequence. This is expected and correct. All 5 node functions are pure in the skeleton: they take an ApplyState, return a partial dict delta, and perform no I/O.
