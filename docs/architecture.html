pi-apply — Architecture

Python LangGraph MCP server exposing the current host handoff surface. load_jd runs to the JD fetch interrupt, and submit_keywords injects host-extracted JDData, runs initial parse/score, and returns tailor context. Sessions persist across server restarts via SqliteSaver at ~/.local/share/pi-apply/apply-sessions.db. Tailoring, rendering, reporting, and finalization are reached through submit_tailor.

Workflow Tool Handler pi_apply/server.py · per-tool guard
Handoff guard before keyword submission: (1) validate compact JDData JSON, (2) verify graph.get_state(config) has a checkpoint, (3) assert snap.next == ("keywords_accept",). LangGraph StateGraph pi_apply/apply_graph.py · build_apply_graph()
StateGraph(ApplyState) with edges jd_fetch → keywords_accept → parse_initial → score_initial → tailor → render → parse_final → score_final → report → finalize. Compiled with interrupt_after=["jd_fetch","keywords_accept"] and SqliteSaver(sqlite3.connect(db_path, check_same_thread=False)). thread_id = session_id — no separate lookup table. Input: validated host-extracted state.keywords. Output delta: no keyword mutation. It verifies JDData is present and preserves the host payload. It does not infer, paraphrase, rank, or generate JD keywords internally. ← triggered by mcp::submit_keywords.

Storage
SqliteSaver — Session Checkpoints ~/.local/share/pi-apply/apply-sessions.db
SqliteSaver(sqlite3.connect(str(db_path), check_same_thread=False)) with explicit .setup() before first use. Persists full ApplyState snapshot after every node execution. Session survives server restart. Missing or out-of-order sessions return typed invalid_session or invalid_state envelopes.
Profile tools · currently bypass graph routing · no session_id required
onboard_user server.py
Accepts resume_content, resume_label, skills, accomplishments, sections. Directly calls the profile node for the walking skeleton and returns next_action="compile_profile".
compile_profile / create_story server.py
Profile compilation and SBI story stubs. The profile graph exists separately, but these MCP handlers still call the target profile nodes directly.
These profile tools return the same JSON envelope shape but do not advance the apply graph. Real profile graph re-entry is intentionally left for a later profile milestone.

"Isolated" means the component can be replaced by editing one file with no cascading edits. The workflow tool handlers in server.py are the orchestration hub — they intentionally coordinate session validation, state injection, and graph invocation in sequence. This is expected and correct. The handoff nodes take an ApplyState, return a partial dict delta, and do not perform hidden keyword extraction.
